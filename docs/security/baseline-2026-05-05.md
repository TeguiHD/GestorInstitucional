# Baseline de seguridad — 2026-05-05

Sistema: Gestor Institucional / Asistencia CSSP.
Datos tratados: estudiantes, asistencia, apoderados, usuarios, comunicaciones y auditoría.

## Marco aplicable

- Chile: Ley 19.628 y reforma Ley 21.719, con entrada en vigencia diferida al 2026-12-01.
- Chile: Ley 21.663 Marco de Ciberseguridad; si el establecimiento o proveedor queda calificado como servicio esencial u operador relevante, preparar reporte de incidentes significativos al CSIRT Nacional según reglas ANCI.
- Referentes de control: NIST CSF 2.0, OWASP ASVS, OWASP Top 10, CISA Secure by Design, MITRE CWE, CIS Controls v8.1, ISO 27001/SOC 2 como gobierno y evidencia.
- Referentes complementarios según cliente/sector: ENISA/NIS2 para ciberhigiene, gestión de incidentes, continuidad, cadena de suministro y vulnerabilidades.

## Controles técnicos mínimos

- Identidad: MFA obligatorio para roles administrativos y directivos; contraseñas Argon2id; bloqueo por intentos fallidos; HIBP k-anonymity habilitado en producción.
- Autorización: ningún registro público de usuarios con roles; creación de usuarios solo por SUPER_ADMIN/DIRECTOR mediante endpoints protegidos.
- Sesiones: access token corto, refresh token rotatorio, detección de reutilización y revocación de familia.
- Red: SSH solo con llave, sin password; UFW deny-inbound con 22/80/443; DB solo en red interna Docker.
- AppSec: validación DTO estricta, whitelist/forbidNonWhitelisted, CORS explícito, Helmet, CSP, rate limit global y auth, logs con redacción de secretos.
- Datos: backups antes de migraciones/importaciones; Prisma migrations como fuente de verdad; RUT único por colegio y matrícula única por curso.
- Auditoría: audit_events con hash chain para acciones críticas; revisar exportaciones, cambios de roles, login y asistencia.
- Dependencias: `pnpm audit --prod` debe quedar sin críticas/altas antes de deploy; riesgo residual documentado si no hay parche compatible.

## Operación de incidentes

1. Clasificar impacto: datos personales, disponibilidad, integridad de asistencia/reportes, credenciales, correo/WhatsApp.
2. Contener: revocar tokens, bloquear cuentas, aislar servicio, congelar exportaciones si corresponde.
3. Preservar evidencia: logs Docker, audit_events, backup DB, hash de artefactos, línea temporal.
4. Notificar: si aplica Ley 21.663, preparar alerta temprana sin datos personales; si hay datos personales, coordinar comunicación a afectados y obligaciones Ley 21.719 al entrar en vigencia.
5. Recuperar: restaurar desde backup probado, rotar secretos, desplegar parche, revisar reglas de detección.

## Pendientes de madurez

- Crear playbook de backup restore probado mensualmente.
- Agregar SAST/dependency audit en CI con bloqueo para critical/high.
- Separar datos de contacto de apoderados desde DOCX/CSV con consentimiento/base legal.
- Revisar `exceljs -> uuid` cuando exista versión compatible o migrar reportes a librería sin advisory.
